# Book storage contract

[`storage/books.mjs`](../storage/books.mjs) is the only app-owned interface for the `books` and `positionOrder` items in `chrome.storage.local`. Viewer, popup, and background code must use its exported asynchronous operations rather than read or write either item directly.

## API

- `getBook(fileUrl)` returns a cloned seven-field record or `undefined`.
- `getPositionTrackingState(fileUrl, viewerId)` is the viewer-only tracked-book read. Under the books lock it validates or initializes the book's order generation, durably registers the 128-bit viewer ID, and returns cloned `{ book, trackingGeneration }`, or `undefined` for a missing book. Registration may write only `positionOrder`.
- `trackBook(fileUrl, { title })` creates the canonical initial record and a fresh empty order generation in one storage write. If the record already exists when its cross-context lock is granted, it validates the relevant order entry, returns the record unchanged, and does not write.
- `listBooks()` returns cloned `{ fileUrl, book }` entries sorted by canonical URL in ascending code-unit order.
- `upsertBook(fileUrl, patch)` creates a record and fresh order generation or patches only `title`, `customTitle`, and `totalPages`. Omitted fields are preserved. New records default to `customTitle: null`, `totalPages: 0`, `currentPage: 1`, and `scrollTop: 0`; `title` defaults to the empty string.
- `updateCustomTitle(fileUrl, customTitle)` sets an existing record's custom title to a string or clears it with `null`, preserving every other field. It returns the updated record, or `undefined` without writing when the record is absent, and never creates a removed or untracked record.
- `hydrateMetadata(fileUrl, { title, totalPages }, { signal })` updates an existing record only while its durable `totalPages` is `0`. It requires a positive page count and preserves every other field even when the preserved `currentPage` is above the actual total, never creates a removed or untracked record, and may be aborted before its write. An already hydrated record is returned without a write.
- `updatePosition(fileUrl, patch, { observedAt }?)` is the documented unordered/legacy position API. It patches one or both position fields, advances `lastReadAt` monotonically, and preserves metadata and all per-viewer high-water marks. A supplied non-negative safe-integer Unix-millisecond `observedAt` makes a strictly earlier persisted second a no-op. A successful write installs a conservative unordered winner barrier at the maximum of the prior winner time and this write's time; it does not erase known stale-viewer rejection state.
- `updatePositionObservation(fileUrl, patch, observation, trackingGeneration)` is the normal service-worker mutation. It atomically updates durable order and, only when the observation wins, the book. It returns `updated`, `stale`, `missing`, or contextual `invalid`. A losing new high sequence can return successful `stale` after writing only its viewer high-water mark; duplicate/lower, wrong-generation, capacity-stale, invalid-future, and missing results do not write.
- `updatePendingPositionObservation(fileUrl, patch, observation)` is the service-worker-only pending-read handoff mutation. Under the same lock, it can apply an observation only when its viewer ID is already registered in the current v2 generation. It never creates order state, migrates legacy state, registers a viewer, or adopts a generation. Missing books return `missing`; absent, legacy, retracked, rotated, duplicate, and unregistered lifetimes return `stale` without writing.
- `removeBook(fileUrl)` returns `true` when it removes a record and its order generation in one write, and `false` without writing when the record is absent.

Book timestamps are non-negative integer Unix seconds and are module-managed. `totalPages: 0` means that the page count is not known yet. All other page counts are positive, `currentPage` starts at 1, and `scrollTop` is a finite non-negative number. A local file can be replaced after metadata hydration, so position updates may advance beyond a stale-low `totalPages` without changing it. Metadata hydration may likewise discover that a replacement PDF has fewer pages than the preserved `currentPage`; persisted positions remain valid state and are clamped only against the currently loaded document by the restore lifecycle. An `upsertBook` mutation that explicitly lowers `totalPages` below the preserved `currentPage` is still rejected.

Inputs are validated before storage access. Keys are canonical, local `file://` PDF URLs with no remote authority. PDF extensions are checked after decoding the pathname, so valid encoded extensions are accepted while malformed escapes and NUL are rejected. Records have exactly the seven fields in [`SPEC.md`](../SPEC.md). A missing `books` item is an empty library; a malformed existing item or record rejects rather than being silently discarded or rewritten. API inputs and results do not alias persisted objects.

`positionOrder` is a separate app-owned map keyed by the same canonical URLs. Current entries have exactly `{ version, generation, winner, viewers }`. `version` is `2`; `generation` is a random 128-bit lowercase hexadecimal tracking-lifetime ID; `viewers` is a map from 128-bit viewer IDs to exact `{ effectiveTime, sequence }` high-water marks; and `winner` is `null` or exact `{ effectiveTime, viewerId, sequence }`. Times are non-negative safe-integer Unix milliseconds. Registered viewers start at sequence `0` and effective time `0`; observed winners and high-water marks use positive safe-integer sequences. An unordered winner uses `viewerId: null` and `sequence: 0`. A viewer winner must exactly match its viewer high-water mark. Every non-null winner must also be the total-order maximum over all positive per-viewer high-water keys; an unordered barrier must dominate every such key. A null winner is valid only while every retained viewer is still at the initial `{ effectiveTime: 0, sequence: 0 }` key.

Missing order state is valid legacy data. The viewer read initializes it, while the first direct observed mutation can adopt its supplied generation after the coarse `lastReadAt` fallback check. The former exact `{ viewerId, sequence, observedAt }` entry is also accepted as strictly validated legacy data and migrated on the next ordering mutation under that same fallback. The fallback applies only while adopting absent or legacy order, not to a durably registered viewer in a current v2 generation; a higher sequence from that viewer can therefore advance position after wall-clock rollback while `lastReadAt` remains monotonic. New tracking and retracking create a fresh generation; an observation from an older generation is therefore stale even when remove and retrack occur in one Unix second. Missing books never acquire order state.

Every operation that can use or change an entry strictly validates the entry for its target canonical URL. A malformed relevant entry rejects without authorizing or rewriting it. Unrelated entries are not treated as authority and are carried through unchanged when an operation must replace the containing map, so a malformed orphan for one URL does not brick valid tracking, upsert, or removal for another URL. A malformed top-level `positionOrder` container still rejects every operation that needs it.

## Concurrency

Chrome documents Storage as an asynchronous bulk key/value API available to all extension contexts; it does not provide a compare-and-swap operation for a nested property. Every mutation here therefore requests the origin-scoped Web Lock named `pdf-resume:books` across its complete read/patch/write sequence. Any mutation that changes both the book and its ordering state issues one `set({ books, positionOrder })` call while holding that lock; a losing new high sequence changes only `positionOrder`. Cooperating app writers therefore cannot interleave a winning order decision and its book position. Web Locks queue the same named exclusive lock across tabs and workers of one origin.

Lock acquisition is bounded at 25 seconds, below Chrome's approximately 30-second inactive extension-service-worker deadline. If the lock is not granted before that bound, the request rejects before its callback runs and storage is not touched. Callers must await every mutation and retry rejected operations when appropriate; starting a mutation without awaiting it can lose the rejection when its context closes.

Popup opt-in uses `trackBook` so two concurrent tracking attempts cannot overwrite the first durable record's title or timestamps. Popup rename uses `updateCustomTitle` so an untrack that wins the lock cannot be undone by a queued stale rename. All app writers must use this module and the Web Locks API must be available. A direct writer for either storage item would violate the serialization contract and can lose concurrent changes. The lock serializes cooperating live contexts, but neither Web Locks nor `chrome.storage.local` provides durable transaction atomicity if a context is terminated during the read-modify-write sequence. Reads may observe either the complete value before or after a completed mutation.

## Viewer position delivery

The viewer never calls `storage/books.mjs` `updatePosition` directly. It creates a random 128-bit viewer ID, then uses `getPositionTrackingState(fileUrl, viewerId)` to register that ID and obtain the current durable tracking generation. Normal debounced saves and final lifecycle snapshots use the private, exactly validated `runtime.sendMessage` protocol in [`shared/position-update-messaging.mjs`](../shared/position-update-messaging.mjs). Every changed snapshot receives a positive per-viewer sequence and non-negative safe-integer Unix-millisecond `observedAt` when it is seen. Messages carry the generation separately. Debounce, retry, document-retirement flush, and lifecycle handoff retain all of that metadata rather than replacing it with delivery or teardown time.

The service worker is the only ordered position writer. It validates the exact message shape, canonical URL, position, observation metadata, optional operation-specific generation, and extension sender before placing both normal updates and pending-read handoffs on one in-worker queue. Future-time validation is contextual and therefore happens under the storage lock: a duplicate/lower sequence or wrong generation is stale first; a strictly higher durably known viewer sequence remains eligible after receiver clock rollback; and a future first/unknown viewer is `invalid` and installs nothing. The worker returns successful `stale` and `missing` outcomes, so controllers do not retry them. Missing books and failed writes do not install state or poison later tracking.

Each viewer's effective time is `max(previous effectiveTime, observedAt)`, so it never decreases when that viewer's wall clock moves backward. A candidate winner key is the lexicographic tuple `(effectiveTime, source rank, viewerId, sequence)`, where unordered barriers outrank observed candidates at an equal time and viewer IDs deterministically order different observed sources. A winner is replaced only by a strictly greater key. This is one transitive total order: same-viewer higher sequences progress at equal normalized time, cross-viewer comparisons cannot create an A/B/A cycle, and a delayed duplicate remains rejected from its durable per-viewer high-water mark after other viewers and worker restarts. A losing higher sequence is still recorded so its later duplicate cannot be reinterpreted as a new viewer. Page direction is never part of the key, and `lastReadAt` remains the maximum persisted Unix second.

An unordered `updatePosition` write preserves viewer high-water marks and replaces the winner with a barrier whose effective time cannot precede the former winner. Thus known lower/duplicate tuples cannot regress it. Because that API has no viewer sequence, a later ordered observation must have a strictly greater total key to replace it; a higher sequence whose clock has not advanced beyond the barrier is conservatively stale.

Different viewers have no shared causal clock. If the receiver clock rolls back, storage can prove that a higher sequence from a durably known viewer is causal, but it cannot distinguish a genuinely delayed unknown viewer from a forged future timestamp; unknown future observations are therefore rejected. Likewise, a previously high effective time can conservatively hold lower-clock cross-viewer observations stale until their times advance. These are deliberate distributed-order limits rather than receipt-time guesses.

Per-book growth is bounded to 64 registered viewer high-water marks. When a 65th distinct viewer performs the viewer-state read, storage rotates to a fresh generation, registers that viewer, discards the prior per-viewer marks, and carries the prior winner time forward as an unordered barrier protecting the preserved book position. Messages from all older-generation viewers then become stale. Rotation is exact and bounded but deliberately retires any older live viewer; such a viewer must be reopened to join the new generation.

While the viewer is live, a rejected normal save retries after 250 ms, 1 second, and 4 seconds. Exhausting those attempts retains the newest snapshot without a pending timer so a later observation or lifecycle event can try it again. The controller's `settled()` waits only for an active write attempt and returns explicit `durable`, `pending`, and `retryPending` state; it does not wait through a scheduled retry or imply durability. Initial tracked-book reads use the same bounded delays, cancel when their document generation retires, and do not retry malformed persisted state. A separate document-scoped metadata lifecycle checks the same canonical record after PDF.js initialization and `pagesinit`; only an existing record with `totalPages: 0` calls the document's cached `getMetadata()` operation and attempts hydration. Its failure is nonfatal and independent of position restore/tracking, while teardown or document replacement aborts any result that has not entered the storage write. The viewer reads with the validated canonical `file://` URL retained by its outer boot wrapper, never PDF.js's blob URL. A present record is restored only after PDF.js application initialization, `pagesinit`, and its `documentinit` initial view; app-owned waits bound initialization, target-page readiness, and the post-restore defense window at 10 seconds each. Restore deliberately does not wait for `pagesPromise` — restore latency scales with the target page, not the page count. After a layout turn it navigates to the clamped saved page, waits for only that page's `pagerendered`, applies the offset, and defends the restored position from that first navigation until handoff: because PDF.js can apply a second initial view after `documentinit` and asynchronous hash/OpenAction destinations can land later, any displacement without genuine interaction is re-asserted — page-only while the target page renders, page and offset once applied — until genuine interaction, `pagesPromise` settling plus one layout frame, or the bounded timeout. A document-scoped listener records exact `PDFPageView` render outcomes from initialization onward because PDF.js marks a failed render `FINISHED`; both a cached failure and a live `pagerendered.error` reject target readiness. The saved page is clamped to the loaded document, and then the canonical viewer container offset is clamped and applied after a layout turn. A lightweight trusted-input lifecycle monitor runs before the tracked-book read: pointer and touch gestures remain correlated through completion, while wheel, key, and click intent has a bounded idle window. Only resulting canonical page/view-area/container-scroll activity makes restoration yield to the live position; restore-owned and other programmatic changes do not. Position listeners are armed once after a final layout turn, and a position changed at that boundary is queued as normal user activity. Missing records remain read-only; malformed state, terminal read/restore failure, and PDF.js initialization rejection or timeout produce a nonfatal accessible app warning while leaving the loaded PDF available.

A `pagehide` handler can synchronously invoke `runtime.sendMessage`, but it cannot synchronously await the returned promise or prove that `chrome.storage.local` is durable before page teardown. Once the tracked-book read has returned a durable generation, genuine changed activity can send one canonical ordered snapshot. While an already-started `getPositionTrackingState` call is still pending, one genuine changed snapshot can instead use the distinct generation-less pending-handoff message keyed by its observation viewer ID. Its worker mutation requests the books lock behind the registration read and succeeds only if that read durably registered the same viewer in the still-current generation; missing, retracked, rotated, or unregistered state is a successful no-write outcome, and the pending mutation cannot register itself. If a read has already failed and is only waiting for its scheduled retry, the viewer deliberately suppresses this handoff because no registration operation is in flight; genuine activity in that narrow failure window can be lost safely rather than authorize an old lifecycle. If an older distinct snapshot is queued or in flight, returning to the pre-flight durable baseline still emits a newer handoff so the older write cannot become final after teardown. A truly durable baseline with nothing older pending, a resolved missing record, absent interaction, and programmatic/default changes send nothing. The worker's open response channel moves asynchronous work out of the dying page and materially improves the handoff, but abrupt browser/process shutdown or service-worker termination can still lose an unfinished mutation. Visibility changes that leave the page alive continue through the awaited normal save path.
